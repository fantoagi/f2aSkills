# cc2Feishu 输入约定

## 1. 对话内容驱动

适用场景：
- “帮我发到飞书”
- “把这篇内容整理成飞书文档并配图”
- “把已有文档某一节补上内容和图”

### 最小必需信息
- 发到哪里：新建还是更新
- 如果新建：标题 + folder/wiki-node/wiki-space 之一
- 如果更新：doc URL / doc token / wiki URL
- 正文主题或草稿

### 推荐补充信息
- 是否需要配图
- 图想表达什么
- 是否已有固定章节结构
- 是否有必须保留的现有文档内容
- 用户最近是否手动修改过当前线上文档
- 本次更新是否只新增内容，还是明确要改旧文章
- 若是给已有栏目追加新文章：目标栏目标题是什么、应该插在该栏目哪一篇文章之后、下一栏目标题是什么
- 若草稿里含有 Mermaid / ASCII 流程 / 结构性代码块：这些代码块是图源而不是正文，发布时应转成 whiteboard

## 2. 本地文件驱动

适用场景：
- 用户已经准备好 markdown 文件
- 用户已有章节草稿或 diagram 文件
- 后续想把一套本地资产稳定复用到飞书

### 常见输入
- `content.md`
- `section.md`
- `diagram.json`
- `board.dsl.json`

### 处理原则
- 优先复用用户已有正文文件，但仅限新内容草稿或参考素材
- 更新已有文档时，当前线上 `docs +fetch` 内容仍是唯一事实源
- 若正文中缺少 whiteboard 占位，根据语义补插
- 若已有 diagram 文件，可直接用于对应 whiteboard；否则默认现场生成

## 3. create / update 判断

### create
当用户说：
- 新建文档
- 发到某个知识库/文件夹
- 把内容生成一篇新的飞书文档

### update
当用户说：
- 更新已有文档
- 在某一节后补内容
- 给已有文章追加图表

更新已有文档时，还要继续判断：
- 是否已经先 fetch 当前线上文档
- 是否属于局部修补模式
- 是否已经进入整段重建模式
- 是否存在必须保留且不可覆盖的对象区域
- 本次更新会不会碰到已有文章结构和正文

## 4. 输出要求

每次执行后，至少向用户返回：
- 文档链接或 token
- 本次采用的是局部修补还是整段重建
- 本次处理的 whiteboard token 列表
- 哪些图已成功回填
- 哪些对象被保留未改
- 发布后校验结果
- 结构验收结论
- 若触发回退，说明触发原因

## 5. 不要做的事

- 不要只插 blank whiteboard 就结束
- 不要把 whiteboard 改成普通图片替代
- 不要在 wiki URL 还没解析前直接当 doc token 用
- 不要把 `docs +fetch` 返回的 `<whiteboard token="..."/>` 当成可写 markdown 回灌
- 不要在发现结构异常后继续叠加局部更新
- 不要默认 overwrite 整篇文档，除非局部修补已不可靠且重建边界明确
- 不要拿旧本地总稿、历史 rebuild 文件或记忆内容覆盖用户线上最新手动修改
