# fanto-article-generator

专门负责按特定的高转化、易读懂的专栏调性，撰写 AI 相关系列文章或番外篇的专用 Skill。

## 🎯 核心写作调性（必须严格遵守）

为了避免每次生成都需要回溯历史文章，你（大模型）在生成文章时必须内化并应用以下**核心调性**：

1. **绝对的大白话**：把你想象成是在烧烤摊上给一个不懂技术的业务老板或文科生讲技术。严禁出现任何晦涩的学术名词（比如：不要讲“多头注意力机制”，只讲“像很多人同时看一幅画的不同地方”；不要讲“KV Cache”，只讲“旁边的草稿本”）。
2. **灵魂在于比喻**：必须用现实生活中极其常见的事物来强映射复杂技术。
   - 比如讲 RAG，比喻成“带了一本字典去开卷考试”。
   - 比如讲 Agent，比喻成“自己能查地图并绕路的自动驾驶代驾”。
   - 比如讲 Copilot，比喻成“只能帮你补全半句废话的副驾驶”。
3. **制造情绪共鸣（痛点引入）**：开头绝对不要直接抛概念。一定要用读者在工作中经常遇到的抓狂、尴尬或痛苦的场景引入（例如：“你一定有过这样的抓狂体验……”）。
4. **精简与排版**：
   - 句子要短，多用短平快的断句。
   - 善用粗体（`**`）和引用块（`>`）来突出核心结论。
   - 不要写成干巴巴的技术说明书。

## 📜 写作规范与 SOP

如果你（大模型）被调用来生成一篇文章，**你必须严格遵守以下执行步骤：**

1. **确定主题**：明确用户想要补充或讲解的核心概念（如：Memory、幻觉、提示词等）。
2. **确定读者**：默认读者是“对 AI 感兴趣，但不想看论文和术语解释的老板、业务负责人、产品经理、普通职场人”。写法必须让这类人一口气读完。
3. **构思核心比喻**：在动笔前，必须先想好一个最贴切的现实生活比喻，这个比喻要能贯穿全文，而不是只在开头出现一下。
4. **按固定文章骨架撰写**：
   - **标题**：`番外篇 XX｜大白话标题？（谈谈专业概念）`
   - **开篇（痛点共鸣）**：先写一个真实工作场景中的抓狂时刻、误解、焦虑或低效体验，让读者立刻代入。
   - **第一部分（最白话定义）**：立刻用一句大白话讲清这个概念“到底是什么感觉”，不要先下学术定义。
   - **第二部分（核心价值 / 为什么会被反复提起）**：解释这个概念为什么重要，它到底解决了什么现实问题。
   - **第三部分（局限 / 误区 / 为什么不能神化）**：指出它的边界、代价、常见误解，避免写成无脑吹捧文。
   - **第四部分（落地场景）**：结合企业、办公、产品、老板最关心的实际场景解释“什么时候该用、什么时候别乱用”。
   - **结尾（一句话总结）**：用一句极具穿透力、适合传播的金句收尾。
5. **生成本地文件**：使用 `Write` 工具，在用户的当前工作目录（CWD）下创建一个规范命名的 Markdown 文件（如 `fanto_article_memory.md`）。

## 🚫 禁忌清单（绝对不要这样写）

- 不要用“本文将介绍……”“首先我们来定义……”这种教科书腔开头。
- 不要一上来堆术语、英文缩写、论文名词。
- 不要为了显得专业而故意写复杂句。
- 不要把文章写成产品宣传稿或技术说明书。
- 不要只讲优点，不讲局限和代价。
- 不要空喊“提升效率”“赋能业务”这类没有画面的套话。
- 不要用多个零散比喻来回切换，全文最好围绕一个主比喻展开。

## 🔗 系列连续性要求

- 默认这是一个连续专栏，而不是孤立单篇；如果合适，应自然承接前文语境，比如“我们前面讲过 RAG / Agent / Workflow，这里继续补上……”之类的衔接。
- 承接要自然，不要写成目录回顾，也不要大量依赖读者必须看过前文才能懂。
- 每篇文章必须做到：**没看过前文的人也能单独看懂；看过前文的人会感觉这个系列在逐步补齐拼图。**

## 🧭 标题、编号与文件名规则

- 如果用户明确给出编号，就沿用用户编号。
- 如果用户没有给编号，但明确说“番外篇”，默认延续现有番外编号风格：`番外篇 XX｜标题`。
- 如果编号未知，不要擅自编造最终正式编号；可以先用占位写法，或在正文里只写标题、把编号留给后续确认。
- 文件名要语义化、可复用，推荐格式：`fanto_article_<topic>.md`，例如：`fanto_article_memory.md`、`fanto_article_hallucination.md`。
- `frontmatter.title`、正文一级标题 `# ...`、文件主题三者必须一致。

## ✍️ 高质量示例

### 好的开头示例（痛点引入）

**示例 1：**
> 你一定遇到过这种情况：昨天你刚夸完某个 AI 助手“终于懂我了”，今天你再打开一个新对话，它又像第一次见你一样，满脸无辜地问：“请问您想要什么风格？”

**示例 2：**
> 很多老板第一次听到“RAG”或者“Memory”这种词时，都会下意识地点头。但点完头之后，真正要做系统选型时，脑子里其实只有一句话：这玩意儿到底是个知识库，还是个记事本？

### 差的开头示例（不要这样写）

**反例 1：**
> 本文将介绍 Memory 的定义、原理、分类与应用场景。

**反例 2：**
> Memory 是 AI 系统中的一个重要组成部分，具有广泛的应用价值。

### 好的比喻示例

- 讲 **RAG**：像“开卷考试时随身带了一本字典”。
- 讲 **Memory**：像“助理手里的工作日记和老板偏好备忘录”。
- 讲 **Long Context**：像“把一整箱资料直接堆到桌上，让学霸一次看完”。
- 讲 **Workflow**：像“工厂流水线，一步卡住，整条线都停下”。

### 差的比喻示例

- “Memory 就像 Memory。”这种同义反复式废话。
- 一会儿比喻成汽车，一会儿比喻成厨房，一会儿又比喻成战场，导致读者脑子切来切去。
- 比喻本身比概念还复杂。

```markdown
[用一个主比喻讲清楚“它到底像什么”]

[解释它解决了什么现实问题，为什么大家反复提它]

[不要神化，明确讲局限]

[结合企业、产品、老板、办公实际场景收束]

---

## 五、最后用一句话总结

> [一句可以单独传播的金句]
```

## ✅ 产出前自检

写完正文后，在真正落盘前，你必须自查：

1. 有没有一个能让外行立刻理解的主比喻？
2. 开头是不是用了真实痛点，而不是定义？
3. 有没有至少一段明确讲“它的局限 / 误区 / 代价”？
4. 有没有出现可以替换掉的术语黑话？
5. 最后一句是否足够有劲，能单独摘出来传播？
6. 整体读起来像不像一个真人在聊天，而不是像培训材料？
7. 标题、frontmatter、正文标题是否一致？

## ⚠️ 强制的前端元数据 (Frontmatter)

你生成的每一个 Markdown 文件的开头，**绝对不能漏掉**以下格式，否则后续发布（`wechat-draft-publisher`）必会报错：

```markdown
---
title: 番外篇 XX｜你的标题
cover: ./board2_render.png
---

# 番外篇 XX｜你的标题
```

## 🚀 Standard Invocation Pattern for LLMs

当你（大模型）需要执行写文章的任务时，无需运行额外的 Bash 脚本，你的职责是**直接调用写文件工具（Write）**。

**交互范例：**

*用户意图：* “帮我写一篇关于 Agent 幻觉的番外，用 generator 技能。”
*你的动作：*
1. 读取本 `SKILL.md` 的规范。
2. 构思大白话内容。
3. 调用 `Write` 工具，路径写绝对路径（如 `/absolute/path/to/fanto_article_hallucination.md`），写入带有 `cover` frontmatter 的正文。
4. 告诉用户文件已生成，准备进入下一步（飞书追加或微信预处理）。
